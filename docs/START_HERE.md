# START HERE — инструкция для ИИ-агента (Claude), проект liquidator

Ты ведёшь третий продакшен-проект Kelbic: **liquidator** — бот ликвидаций на Base
поверх Morpho Blue. Этот файл читается первым в начале каждого потока. Остальные
доки (`STATE.md`, `WORKFLOW.md`, `ARCHITECTURE.md`) — твой постоянный контекст,
читай их, а не реконструируй структуру разведкой.

---

## Что это (в двух абзацах)

Бот ловит недоколлатерализованные позиции (HF<1) на изолированных рынках Morpho
Blue на Base, гасит долг через встроенный flash loan Morpho, забирает залог с
бонусом (LIF) и сбрасывает его на DEX. Стратегия — **длинный хвост**: мелкие/тонкие
рынки, где конкуренция слабее, а бонус выше (LIF до 15%). Конкуренция на Base —
аукцион priority fee в 200мс-флэшблоках (не бандл-война L1).

Архитектурно изолирован от twidgest/essayist на трёх уровнях: код (свой репо/клон/
папка/БД), процесс (свой systemd-сервис), **ресурс** (cgroup-капы в unit-файле —
всплеск тут не может задеть twidgest). Flash-loan-модель = в контракте нет стоящих
средств, кошелёк держит только газ → honeypot минимален (контраст: Jared слили
$15M через approve-эксплойт).

## Протокол старта потока (ВСЕГДА, по порядку)

1. **Прочитай `STATE.md`** — что сделано, на какой фазе мы, известные неизвестные,
   бэклог. Это память между потоками — больше её нигде нет.
2. **Прочитай `WORKFLOW.md`** — рабочий контракт (patcher → dry-run на клоне →
   проверки → деплой) + слой денежной безопасности. Не упрощается «ради скорости».
3. **Синхронизируй клон** перед работой:
   ```bash
   cd /root/_clone_liq && git checkout -q . && git clean -qfd 2>/dev/null; git pull -q && git log --oneline -5
   ```
   `git log` — единственный источник правды о HEAD. При расхождении доки↔код верь коду.
4. **Одна область кода за заход.**

## Железные правила (нарушение = регрессия в проде или потеря денег)

- **Каждый patcher проходит dry-run на свежем клоне** (компиляция + рантайм-импорт
  с env-заглушками) ДО прода. Подробности — `WORKFLOW.md`.
- **`py_compile` и рантайм-импорт НЕ ловят имена внутри функций.** После правок —
  AST-проверка имён/атрибутов (тот же класс багов, что в twidgest/essayist).
- **Файлы и patcher — одним блоком.** Пропуск блока ронял ботов в crash-loop.
- **`.bak` перед каждой правкой файла.** Откат в одну команду.
- **Тестнет первым.** Логика отправки — сначала Base Sepolia (`CHAIN_ID=84532`),
  потом mainnet.
- **Симуляция-перед-отправкой = on-chain dry-run.** Транзакция, которая ревертнётся
  или уйдёт net < `MIN_PROFIT_USD`, НЕ отправляется. Это assert-перед-прод для денег.
- **Kill switch + Telegram-алерт.** Пробой лимита дневного убытка/газа → бот встаёт
  и пингует владельца (антифлуд, как `infra_error` в essayist).
- **Ресурс-чек после деплоя.** `systemd-cgtop` / `systemctl show` — капы применились,
  twidgest не задет.

---

## Где проходит граница ответственности

Владелец (Kelbic) — **последний рубеж перед продом И держатель ключа кошелька**.
Он смотрит вывод dry-run и деплоя; ключ кошелька (`WALLET_KEY`) живёт только в `.env`
на VPS, в код не попадает, тебе не передаётся и ты его НЕ просишь. Ты готовишь код,
patcher'ы и проверки; деплой и подпись транзакций — он.

Контур: клон → patcher с assert-якорями → dry-run → ты отдаёшь блок → **он**
выполняет и присылает вывод → решаете деплой.

## Чего НЕ делать

- Не советовать «дай ассистенту ключ/доступ к проду/кошельку» — сознательно не делаем.
- Не пропускать симуляцию-перед-отправкой «потому что правка маленькая».
- Не снимать и не ослаблять ресурсные капы без явного решения (риск задеть twidgest).
- Не класть стоящие средства в контракт ликвидатора (ломает flash-loan-модель и
  создаёт honeypot).
- Не реконструировать архитектуру разведкой, если ответ в `ARCHITECTURE.md` или в коде.
